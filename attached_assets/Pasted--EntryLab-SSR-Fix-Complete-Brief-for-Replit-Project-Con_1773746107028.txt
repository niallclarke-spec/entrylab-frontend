# EntryLab SSR Fix — Complete Brief for Replit

---

## Project Context

**EntryLab** (entrylab.io) is a forex broker and prop firm review/news aggregator.

### Tech Stack
- **Frontend:** React SPA built with Vite (outputs `index.html` + hashed JS/CSS bundles in `/assets/`)
- **Backend:** Node.js/Express server with PostgreSQL database
- **Content API:** REST API at `/api/articles`, `/api/brokers`, etc. — returns full article content, broker data, metadata from PostgreSQL
- **Hosting:** VPS on Hostinger
- **NO WordPress.** All content lives in PostgreSQL and is served via the app's own REST API. There is a legacy `admin.entrylab.io` WordPress install that should be completely ignored — it is being decommissioned.

### How the Server Currently Works

The Express server currently does TWO things well:

1. **Serves API routes** (`/api/*`) — these return JSON from PostgreSQL with full content
2. **Serves the SPA** — for ALL non-API routes, it reads `index.html` from the Vite build, injects per-page `<title>`, `<meta>` tags, Open Graph tags, Twitter Card tags, JSON-LD structured data (Article, BreadcrumbList, Organization), and canonical URLs into the `<head>` — then serves this modified HTML

The server already resolves URL slugs to database content for meta tag injection. **It just doesn't inject the actual body content.**

---

## The Critical Problem

**The HTML body contains ZERO content.** Every page serves the same empty shell:

```html
<body>
  <noscript><!-- GTM iframe --></noscript>
  <div id="root">
    <!-- Only skeleton loaders (animated placeholder blocks) -->
    <!-- NO article text, NO headings, NO paragraphs -->
  </div>
</body>
```

All article content only appears AFTER JavaScript executes in the browser. This means:

- **Google** partially indexes pages (sees meta tags but struggles with JS-rendered body)
- **AI crawlers** (ClaudeBot, GPTBot, PerplexityBot, etc.) get **nothing** — the site is invisible to AI search
- **Social previews** work (OG tags are in `<head>`) but actual content is unreachable
- The site has ~40+ published articles across broker guides, broker news, prop firm news, trading tools — none have readable body content for any crawler

### Proof

```bash
# Returns 0 — no <p> tags in the body at all
curl -s https://entrylab.io/broker-guides/hero10x-review | grep -c "<p>"

# The <head> IS correct — title, meta, OG, JSON-LD all present and per-page
curl -s https://entrylab.io/broker-guides/hero10x-review | grep "<title>"
# → HeroFX Funded Account (Hero10X): Trade with 10× Capital ✅
```

## What Needs to Happen

### The Fix: Inject Full Content into `<body>` Server-Side

The server already:
1. Intercepts all non-API requests
2. Reads the Vite-built `index.html` template
3. Resolves the URL slug to a database record
4. Injects `<title>`, meta tags, OG tags, JSON-LD into `<head>`

**Now it also needs to:**

5. **Fetch the full `content` field** from the same database query it's already making
6. **Inject the article HTML into `<body>`** inside `<div id="root">`

### Where to Inject

Currently the body looks like:
```html
<div id="root">
  <!-- Skeleton loaders only -->
  <div style="min-height:100vh;...">
    <header><!-- skeleton navbar --></header>
    <main><!-- skeleton cards --></main>
  </div>
</div>
```

Change it to:
```html
<div id="root">
  <!-- SSR content — visible to crawlers, replaced by React on hydration -->
  <div id="ssr-content" class="prose" style="max-width:768px;margin:0 auto;padding:32px 24px;">
    <h1>{article.title}</h1>
    <p style="color:hsl(var(--muted-foreground));margin-bottom:2em;">
      Published {formatted date} · By {article.author} · {article.categoryName}
    </p>
    {article.content}
    <!-- Full HTML from the content field -->
  </div>

  <!-- Existing skeleton stays as visual fallback -->
  <div style="min-height:100vh;...">
    <header><!-- skeleton navbar --></header>
    <main><!-- skeleton cards --></main>
  </div>
</div>
```

When React mounts, it replaces the entire `#root` innerHTML — so the SSR content naturally disappears once the app hydrates. No special cleanup needed.

## Implementation Approach

### Extend Existing Express Server

The server already has the slug resolution and meta injection logic. Find the route handler that catches all non-API requests (likely something like `app.get('*', ...)`) and extend it:

```javascript
// Pseudocode — adapt to match your existing codebase
app.get('*', async (req, res) => {
  let html = readFileSync('dist/index.html', 'utf-8');
  
  const { slug, category, routeType } = parseRoute(req.path);
  
  if (routeType === 'article' && slug) {
    // Fetch article from PostgreSQL (you're ALREADY doing this for meta tags)
    const article = await getArticleBySlug(slug);
    
    if (article) {
      // Inject <head> tags (ALREADY HAPPENING — keep this)
      html = injectMetaTags(html, article);
      
      // NEW: Inject body content
      const ssrContent = `
        <div id="ssr-content" class="prose" style="max-width:768px;margin:0 auto;padding:32px 24px;">
          <h1>${escapeHtml(article.title)}</h1>
          <p style="color:hsl(var(--muted-foreground));margin-bottom:2em;">
            Published ${formatDate(article.publishedAt)} · By ${article.author} · ${article.categoryName}
          </p>
          ${article.content}
        </div>
      `;
      
      // Insert SSR content at the beginning of #root, before skeleton
      html = html.replace(
        '<div id="root">',
        `<div id="root">${ssrContent}`
      );
    }
  } else if (routeType === 'broker' && slug) {
    // Fetch broker profile from PostgreSQL
    const broker = await getBrokerBySlug(slug);
    
    if (broker) {
      html = injectMetaTags(html, broker);
      
      const ssrContent = `
        <div id="ssr-content" class="prose" style="max-width:768px;margin:0 auto;padding:32px 24px;">
          <h1>${escapeHtml(broker.name)} Review</h1>
          <p style="color:hsl(var(--muted-foreground));margin-bottom:1em;">${escapeHtml(broker.tagline)}</p>
          ${broker.content}
        </div>
      `;
      
      html = html.replace(
        '<div id="root">',
        `<div id="root">${ssrContent}`
      );
    }
  } else if (routeType === 'listing') {
    // Category listing page — inject article list with titles, excerpts, links
    const articles = await getArticlesByCategory(category);
    const ssrContent = buildListingHTML(articles, category);
    html = html.replace('<div id="root">', `<div id="root">${ssrContent}`);
  }
  
  res.send(html);
});
```

### Caching

Add in-memory caching (or a simple Map with TTL) to avoid hitting PostgreSQL on every page load:

- Cache rendered HTML per URL path for 5-10 minutes
- Invalidate on content update (or just let TTL expire)
- This is especially important for listing pages that query multiple articles

```javascript
const cache = new Map();
const CACHE_TTL = 5 * 60 * 1000; // 5 minutes

function getCached(key) {
  const entry = cache.get(key);
  if (entry && Date.now() - entry.time < CACHE_TTL) return entry.html;
  return null;
}

function setCache(key, html) {
  cache.set(key, { html, time: Date.now() });
}
```

### HTML Safety

The `content` field from PostgreSQL contains HTML that was authored in your admin tools. It should be safe to inject directly. If you want extra safety, run it through `sanitize-html` on the server, but this is optional since you control the content.

### Image URLs

Some `featuredImage` URLs still reference `admin.entrylab.io/wp-content/uploads/...`. These work as-is since they're absolute URLs. Long-term, migrate these to your own storage, but for now they'll render fine.
